## 0. 先纠正一个关键理解：GRPO 不是挑最大候选

RLVR-World / 当前 `dor.grpo` 的流程是：

$$
q=(s,a)\rightarrow \pi_\theta(\cdot|q)\ \text{sample }G\text{ candidates }o_1,\ldots,o_G
\rightarrow R_i \rightarrow
A_i=\frac{R_i-\mathrm{mean}_j R_j}{\mathrm{std}_jR_j+\epsilon}
$$

然后策略梯度近似为：

$$
\mathcal L_{\mathrm{PG}}=-\frac1G\sum_{i=1}^{G}A_i\log \pi_\theta(o_i|q).
$$

含义：

- **不是**只挑 reward 最大的候选做 SFT。
- 是**所有候选都进更新**：组内高于均值的候选 \(A_i>0\) 被提高概率，低于均值的候选 \(A_i<0\) 被压低概率。
- 因而本文真正关心的是 reward 在同组候选里的**相对排序 / 优势符号 / 间隔可靠性**，不是绝对分数。

这也是为什么“奖励噪声地板腐蚀组内排序”能直接打中 GRPO。

## 1. 对当前“多粒度、多域、多指标 reward 融合”想法的审稿人判断

这个方向可以保留，但不能直接作为主贡献宣称“指标越多越好”。现有负结果已经说明：

- `dorw` 地板加权融合最后塌成 `code`；
- `hybrid_tok` 低地板融合也没有超过单分量；
- Dr.GRPO / hard floor-filter 没有稳健收益；
- FD-DINO/KID 初步显示 `code` 在分布特征评测上不赢 `pixel/pixel_tok`。

所以可写的不是“我们堆了更多指标”，而是：

> 多域指标只有在它们提供**正交且可靠的组内排序信息**时才有用；否则融合只是把噪声或冲突排序混进 GRPO。

换句话说，reward 设计必须从“多”改成“可验证可靠”。论文可保留多粒度/多域作为一个 framework，但主方法需要有清晰的可靠性判据。

## 2. 建议贡献链（当前最稳版本）

### C1. 问题定义：tokenized RLVR 的 reward noise floor

定义：

$$
\phi_{\mathrm{tok}}^{(d)}
=
\mathbb E_{s'}\left[
d\left(\mathrm{decode}(\mathrm{encode}(s')),s'\right)
\right].
$$

注意写法：

- 这是我们定义的术语，不冒充已有概念。
- 地板是 \((tokenizer, d)\) 对的性质，不夸大成所有指标都同等严重。
- FID/FVD/FD-DINO 是评测集合统计，不能直接作为逐候选 RLVR reward。

### C2. 机理：GRPO 消费组内排序，地板造成排序翻转

把奖励写成：

$$
R_i=R_i^\star+\eta_i.
$$

若含噪奖励与干净奖励的相关为 \(\rho\)，高斯近似下 pairwise ordering flip rate 为：

$$
P_{\mathrm{flip}}=\frac{\arccos(\rho)}{\pi}.
$$

这是目前最硬的理论资产。它把 C1 从“现象”推进到“为什么会伤 GRPO”。

### C3. 方法：可靠性校准的可验证奖励，而非简单多指标融合

推荐把方法口径从“DoR=code 一招”升级为：

> Reliability-calibrated verifiable reward for tokenized video world models.

包含两个可落地实例：

1. **floor-cancelled target alignment**：`code` 与 `pixel_tok`，把奖励对齐到 tokenizer 可达目标，减少 decoder floor。
2. **rank-reliability weighting**：根据候选间 reward gap 与噪声尺度，软降权易翻转候选的 advantage。

如果第 2 点 pilot 成功，它就是新的第二方法贡献；如果失败，它仍是一个强消融，证明关键在 reward 目标而不是 GRPO trick。

## 3. 三条候选方法的取舍

### A. 秩可靠性软加权（优先 pilot）

不要硬删组，也不要在 reward 里做组内二次归一化。对已经得到的 advantage 做候选级软权重：

$$
c_{ij}=2\Phi\left(\frac{|R_i-R_j|}{\sqrt{2}\sigma_\eta}\right)-1,
$$

$$
w_i=\min_{j\neq i}c_{ij},
\qquad
\tilde A_i=w_iA_i,
\qquad
\tilde A_i\leftarrow \tilde A_i-\frac1G\sum_j\tilde A_j.
$$

实现点：

- `src/dor/rewards.py`：新增 `rank_reliability_weights`。
- `src/dor/grpo.py`：`shape_advantage` 后乘权重并 re-center。
- `scripts/train_grpo.py`：新增 `mode=rankrel`、`--rank_weight`、`--rank_sigma`、`--rank_sigma_scale`、`--rank_min_weight`，默认关闭。

Pilot：

- arms：`pixel`, `pixel+relw`, `pixel_tok`, `pixel_tok+relw`, `code`, `code+relw`。
- seeds：先 5；若正向扩到 8-10。
- 判据：flow + FD/KID/PRDC；必须配对多 seed 稳超原臂。

失败模式：

- reward gap 本身被噪声拉大，权重误信 outlier；
- 中段候选被压太多，探索变差；
- 若不 re-center，会引入整体 log-prob 偏置。

### B. 分布 witness reward（暂缓，等 PRDC 结果）

FID/KID/FD-DINO 不能伪装成单样本 reward。可行的是 MMD witness：

$$
r_{\mathrm{mmd}}(x_i)
=
\frac{2}{|Y|}\sum_{y\in Y}k(f(x_i),f(y))
-
\frac{1}{|X|-1}\sum_{j\neq i}k(f(x_i),f(x_j)).
$$

其中 \(X\) 是当前生成候选集合，\(Y\) 是真实特征库。

只能作为小权重 regularizer：

$$
R_i=R_i^{\mathrm{FR}}+\lambda r_{\mathrm{mmd}}(x_i).
$$

审稿风险：

- 它优化边际真实感，可能偏离“这一个动作条件下的真实下一帧”；
- 若用 DINO witness 又用 FD-DINO 评测，容易被质疑 home advantage；
- batch 小时 witness 方差大。

触发条件：

- 只有当 PRDC 明确显示 `code` 或 `pixel_tok` 高 precision、低 recall 时，才值得试。

### C. 运动残差对齐奖励（第三条，建议并行 pilot）

如果要补一个真正“正交”的 reward，不要再堆 LPIPS/MSE/code 的同类保真信号，而是显式对齐 motion residual：

$$
\Delta z_i=z_i-z_t,\qquad
\Delta z'=z'-z_t.
$$

不能用 \(\| \Delta z_i-\Delta z'\|\)，因为它退化为 \(\|z_i-z'\|\)。应使用方向与幅度：

$$
R_i^{\mathrm{dyn}}
=
\cos(\Delta z_i,\Delta z')
-
\gamma
\left|
\log\frac{\|\Delta z_i\|+\epsilon}{\|\Delta z'\|+\epsilon}
\right|.
$$

再加 motion gate：

$$
R_i=R_i^{\mathrm{base}}+\lambda\mathbf 1[\|\Delta z'\|>\tau]R_i^{\mathrm{dyn}}.
$$

实现点：

- `src/dor/grpo.py` 需把当前帧 `cur_idx` 传给 reward 函数。
- `src/dor/reward_spaces.py` 新增 `code_dyn` / `pixel_tok_dyn`。

Pilot：

- arms：`pixel`, `pixel_tok`, `code`, `pixel_tok_dyn`, `code_dyn`。
- 判据：flow 稳超对应 base，FD/KID 不崩，LPIPS 不明显退化。

失败模式：

- FSQ code delta 不一定线性表示 motion；
- 静态窗口太多导致有效样本少；
- 方向对了但内容错，LPIPS/FD 下降。

## 4. 下一轮最小实验矩阵

优先只跑两组，不再大面积扫：

### Pilot 1：A 是否能把 C2 理论变成方法

```bash
python scripts/train_grpo.py \
  --rewards pixel,pixel_tok,code \
  --modes gt_only,rankrel \
  --seeds 0,1,2,3,4 \
  --steps 150 --K 16 \
  --out_dir outputs/grpo_v8_rankrel
```

`rankrel` 是 mode：先按普通 GRPO 得到 \(A_i\)，再乘候选级 rank-reliability weight 并 re-center。`gt_only` 是同 reward 的原始对照。
默认 `--rank_min_weight 0.05`，避免全部 gap 都小的时候把有效学习率压到近零。

敏感性可另跑：

```bash
python scripts/train_grpo.py \
  --rewards pixel,pixel_tok \
  --modes rankrel \
  --seeds 0,1,2,3,4 \
  --steps 150 --K 16 \
  --rank_sigma_scale 0.5 \
  --out_dir outputs/grpo_v8_rankrel_s05
```

### Pilot 2：C 是否提供真正正交动态信号

```bash
python scripts/train_grpo.py \
  --rewards pixel_tok,code,pixel_tok_dyn,code_dyn \
  --modes gt_only \
  --seeds 0,1,2,3,4 \
  --steps 150 --K 16 \
  --dyn_lambda 0.25 --dyn_gamma 0.25 \
  --out_dir outputs/grpo_v9_dyn
```
